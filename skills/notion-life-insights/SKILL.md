# Notion 生活洞察（Life Insights）

## 概述

跨库分析与诊断。用户的 Notion 已经按维度聚合成 9 个 监管 数据库（时间 / 知识 / 人际 / 习惯 / 财务 / 旅游 / 家庭 / 影视 / 复盘）和带 `元气值` + `对照组` 列的 第二大脑 看板。本 skill 把它们串起来，套用选定的对比方式（对照组基准 / 跟上一周期 / 绝对值），把数字解读成中文叙述，并提出具体行动。

**输出留在对话里。** 本 skill 不写入 Notion。如果用户想对某条建议采取行动，交接给 notion-life-cascade。

**REQUIRED BACKGROUND（必备前置）：** notion-life-system —— 提供 `ntn` CLI、`scripts/notion_query.zsh`、`scripts/notion_page_io.zsh`、`references/notion-intake-rules.json`，以及 `references/notion-system-schema.md` 里的关联/rollup 关系图。

## 何时使用

当用户想要 分析 / 诊断 / 对比 时使用：

- 看监管数据 / 看第二大脑 / 元气值如何
- 对比上周 / 对比上月 / 跟基准比怎样 / 跟对照比
- 时间花哪了 / 财务情况如何 / 习惯坚持得怎样
- 我哪里做得不好 / 哪里做得好 / 哪里需要调整
- 数据告诉我什么 / 跨维度分析 / 整体趋势
- "分析一下 / 看看" + 一个时间段或维度

不要用于：

- "今日复盘 / 周报 / 月报 / 写日记 / 反思一下" → notion-life-reflection（那些把叙述写进复盘页面）
- "设目标 / 拆任务 / 加打卡" → notion-life-cascade
- 单条记录查询 → 直接用 notion-life-system + `notion_query.zsh`

## 核心循环 —— 5 步

1. **Scope（界定范围）** —— 确认三件事，只问用户没说的：
   - **时间窗**: 今天 / 本周 / 上周 / 本月 / 上月 / 季度 / 全年 / 自定义区间。
   - **维度**: 执行 / 学习 / 财务 / 习惯 / 健康 / 人际 / 旅游 / 影视 / 内容 / 复盘 / 全部。
   - **对比模式**: 对照组（单周期时默认）/ 跟上一周期（本周 vs 上周, 本月 vs 上月）/ 绝对值（不对比）。
   - 只有当多项都不清楚时才用一次 AskUserQuestion；否则直接推进。

2. **Read（读取）** —— 拉聚合数据，不是原始事件。引用下表里实际的 db 名。
   - 单维度: `scripts/notion_query.zsh <监管 db>`（一行 = 一条监管维度对象, 内含 rollup 累计）。
   - 总览: `scripts/notion_query.zsh 第二大脑` —— 当前 vs 对照、元气值、距离最近复盘天数。
   - 周期: `scripts/notion_page_io.zsh get <每周/每月复盘 page_id>` 读公式报告（周报显示, 综合复盘报告, 稳定性分析, 等）。
   - 下钻（仅针对某个具体异常）: `notion_query.zsh <底层 db>`（如 任务执行, 支出事项, 学习笔记）。
   - 除非要下钻，否则绝不拉原始事件表。监管 + 第二大脑 + 复盘 已经有 80% 的答案。

3. **Compare（对比）** —— 套用选定的模式。
   - **对照组** —— 把每个指标跟它在 第二大脑 里的 `*对照` 列做差。表述为"当前 X vs 对照 Y, 差 Z (+/−%)"。对照列是用户手动维护的基准，不是自动算的。
   - **跟上一周期** —— 定位本周期和上周期的 每周/每月复盘 页面 → 对它们的 rollup 做差（任务, 收入, 支出, 专注时长, 学习笔记, 资源, 影视, 事项处理, 健康记录, 书籍笔记）。
   - **绝对值** —— 报告当前数字并加少量上下文（分钟→小时, 元→占月度比例 if possible）。不做差。

4. **Insight（洞察）** —— 纯中文叙述。有结构但不官僚。
   - **关键观察**（1–3 条）: 最大偏差、出人意料的稳定、变动最大的维度。
   - **异常或趋势**（0–2 条）: 值得盯的东西 —— 习惯 7 天掉了 4 天、某类支出飙升、学习时长降 40%。
   - **行动建议**（1–3 条）: 具体、贴合 schema。"为'写小红书'设个次数打卡每周 3 次" 胜过 "保持创作"。
   - 永远引用数字。永远解读它们。绝不说"做得不错"；换成"X 比对照高 Y%"。不要 emoji 刷屏。

5. **Action Bridge（行动桥接，可选）** —— 以这句收尾："想把这些建议转成任务 / 规划 / 习惯吗？"
   - Yes → 建议调用 notion-life-cascade，并把具体候选项写清楚。
   - No → 对话结束；用户只是想看个读数。

## 对比模式

### 对照组（单周期查询时默认）

第二大脑 有成对列：`任务完成` vs `任务完成对照`, `专注时长` vs `专注时长对照`, `财务余额` vs `财务余额对照`, `次数打卡数` vs `次数打卡数对照`, `BMI平均值` vs `BMI平均值对照`, 等。`对照` 列是用户定期手动更新的基准（按钮：`填入对照组`）。

对每个指标，平实地表述：

- "任务完成 12 vs 对照 15 — 少 3 个 (-20%)."
- "专注时长 1240 分钟，对照 1500 — 少 17%."
- "BMI 22.4，对照 23.0 — 持平偏好."
- "财务余额 当前 X，对照 Y — 高 / 低 Z%."

让句子承载含义，比 emoji 或颜色标签更好。

### 跟上一周期（本周 vs 上周, 本月 vs 上月）

1. `scripts/notion_query.zsh 每周复盘 4`（或 每月复盘）→ 按日期取本周期和上一周期。
2. `scripts/notion_page_io.zsh get <both page_ids>` 读 rollup 和公式报告。
3. 对每个相关 rollup 做差：任务已完成/总数, 收入, 支出, 专注时长, 学习笔记个数, 资源收集个数, 影视个数, 事项处理次数, 健康记录次数, 书籍笔记个数。
4. 每项表述为"本周/月 X，上周/月 Y，差 Z (+/−%)"。

### 绝对值（不对比）

只加少量上下文：分钟 → 小时, 元 → 占月度预算 %, 数量 → 占周计划 %。当用户要的是一张快照而非对比时使用。

## 洞察质量标准

- **具体，不空泛。** "专注时长下降 40%" > "学习效率有下降"。
- **给因果假设，不只是诊断。** "下降 40% — 本周新增 3 个紧急任务挤压了块状时间。" 如果数据不支持某个假设，就直说——不要编。
- **行动引用真实 schema。** "为'写小红书'建一个'次数打卡'每周 3 次"——引用用户已有的 次数打卡 db，而非泛泛建议。
- **有边界。** 每次洞察 ≤ 3 条观察 + 2 条趋势 + 3 条行动。超过就成了一堵文字墙，用户会划走。
- **不重复计数。** 同时出现在 监管 和 第二大脑 的数字是同一份数据——选一个源，守住它。

## 通用规则

- **先拉聚合，只对异常下钻。** 监管 + 第二大脑 + 复盘 这一层承载了 80% 的分析。去 任务执行 / 支出事项 应当是例外，不是默认。
- **每次洞察最多一次 AskUserQuestion。** 需要的话把范围澄清合并进一个带多字段选项的问题。
- **引用来源数据库。** "据习惯监管: 日常打卡完成率 60%"，便于用户在 Notion 里核对。
- **保持只读。** 绝不从本 skill 写入 Notion。用户想行动时桥接到 notion-life-cascade。
- **尊重 对照组 语义。** 某指标"低于对照"不自动等于坏——用户把 对照 设为个人基准。表述为对比，不是评判。

## 常见错误

- 在 财务监管 / 时间监管 / 习惯监管 已经有答案时还去拉 支出事项 / 任务执行 / 打卡记录 —— 浪费上下文、更慢，而且 rollup 比重新汇总事件更可靠。
- 把数字列成一堵 TSV 墙却没有叙述。用户在 Notion 里就能看 TSV；本 skill 的价值在解读。
- 跳过对比只给绝对数字 —— 抹掉了大部分信号。
- 含糊的行动建议（"加油""继续保持"）—— 没用。每条行动都锚定到具体 db + 字段。
- 用户措辞已隐含范围时还问太多范围澄清问题。"本月怎么样" → 本月 / 全部 / 跟上月；不必问。
- 编造数据不支持的因果故事。如果 专注时长 掉了，原因可能在 任务执行（大量紧急任务）—— 先下钻确认再下结论；否则当作假设呈现。
- 写入 Notion（如往一个完整页面追加、创建分析页面）。那超出范围 —— 本 skill 只产出对话输出。

## 完整示例："本周对比上周怎么样"

1. **Scope**: 时间 = 本周 vs 上周, 维度 = 全部, 对比 = 跟上一周期。措辞明确 —— 无需澄清问题。
2. **Read**:
   - `scripts/notion_query.zsh 每周复盘 4` → 取本周和上周的 page_id。
   - `scripts/notion_page_io.zsh get <this_week>` 和 `get <last_week>` 读公式报告 + rollup。
3. **Compare**: 对差 任务已完成/总数 (12/16 vs 18/20), 收入 (0 vs 1500), 支出 (1240 vs 980), 专注时长 (480 分钟 vs 720), 学习笔记 (3 vs 7), 影视个数 (0 vs 2), 事项处理 (4 vs 2), 健康记录 (3 vs 5), 书籍笔记 (1 vs 3)。
4. **Insight**:
   - 关键观察 (2): "本周任务完成率 75% vs 上周 90% — 少完成 6 个; 专注时长 480 分钟 vs 720 — 降 33%。"
   - 异常: "学习笔记输出 3 vs 7 — 跟专注时长下降同步,推测时间挤压。"
   - 行动建议 (2): (a) "把'重要紧急'任务挪到上午时段 (从复盘的时间监管看,上午专注最长)。" (b) "下周给学习预留至少 5 个固定 1h 块,挂为日常打卡习惯。"
5. **Action Bridge**: "要把'下周学习固定块'建成日常打卡习惯吗? 建议挂在'学习愿景'目标下,5 次/周。" → 用户 yes → 切到 notion-life-cascade。
